# domain-design.md 格式规范

此文件定义 `domain-design.md` 的输出格式

## 核心原则

- **引擎无关** 不出现 Screen、Label、Node、Signal、@export、GDScript 等引擎概念。
- **分模块编写** 从需求中识别功能模块，每个模块独立完成领域分析三件套。
- **包含边界情况清单** 边界情况清单是领域模型的核心价值——引擎 API 不会提醒你漏了边界，但领域模型会。

## 模板

```markdown
# 领域设计

## 模块划分

从需求文档中识别出 {M} 个可独立建设的模块：

| 模块 | 可独立交付的内容 | 包含的行为 |
|------|-----------------|-----------|
| {模块 A} | {一句话——这个模块作为一个整体交付什么功能} | 行为 #1, #2, #3, #4, #5 |
| {模块 B} | {一句话} | 行为 #6, #7 |


## 模块详情

### {模块 A}

**这是什么模式：** {状态机 / 资源管理 / 事件队列 / 空间查询 / 命令模式 / 观察者 / 数据驱动 / ...}

**业界通用做法：**

{这个模式在游戏开发中一般怎么实现。讲清楚核心概念、状态/阶段、关键规则。用引擎无关的语言。}

**核心规则：**
- {规则 1}
- {规则 2}

**边界情况清单：**

| # | 边界情况 | 预期行为 |
|---|---------|---------|
| 1 | {情况} | {行为} |
| 2 | {情况} | {行为} |

### {模块 B}

**这是什么模式：** ...

**业界通用做法：** ...

**边界情况清单：** ...

> 如果某个模块的领域模式很明显（如"显示一个静态文本"→ 无复杂领域逻辑），简写即可，不强行拔高。
```

## 内容规则

**必须遵守：**
- 先划分模块，再对每个模块做领域分析三件套（模式 → 业界做法 → 边界）
- 领域描述不出现引擎概念
- 边界情况清单至少覆盖：输入为空、容量上限、并发冲突、中断/取消

**禁止内容：**
- 引擎术语（Screen、Label、Node、Signal、@export、GDScript 等）
- 代码符号（class 名、方法名、变量名）
- 实现细节
