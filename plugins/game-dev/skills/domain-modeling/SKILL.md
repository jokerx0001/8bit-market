# 领域建模

将需求文档翻译为具体引擎技术无关的领域模型。**铁律：只做分析和输出文档，不写实现代码，不涉及引擎概念。**

**格式契约：** `references/domain-design-format.md` 定义了 domain-design.md 的完整格式规范。所有输出必须遵守。

## 参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `--from {task_dir}` | 是 | 任务目录路径（如 `.godot-dev/feat-3`） |
| `--tech {tech}` | 建议 | 技术栈标识（renpy / godot）。未传入时从项目 CLAUDE.md 检测 |
| `--auto` | 可选 | 跳过人工确认 |

## 前置步骤：确定 dev_dir

1. 如果传入了 `--tech`，直接使用。否则 grep 项目 CLAUDE.md 检测技术栈关键词。
2. 读取 `${CLAUDE_PLUGIN_ROOT}/references/{tech}/config.md`，从 `## 产物目录` 节提取 `dev_dir`。

## 领域建模是什么

对每个独立模块回答：**这个模块应该用什么领域模型来建设？**

```
模块: 换弹
  ├── 这是什么模式？ → 有限状态机（IDLE → RELOADING → READY）
  ├── 业界通用做法？ → FSM + Timer 驱动 + 事件通知
  └── 有哪些边界？   → 满弹不换、中断回滚、空弹自动触发
```

输出的是**建设方案**：这个模块在领域层的最佳实践是什么。不和引擎绑定——架构设计拿到后翻译成具体引擎的落地方案。不同模块可能独立使用同一领域模式，这是正常的。

## 为什么领域模型要先于架构设计

- 领域设计让你先想清楚"做什么"，再决定"怎么写"——顺序不能反
- 引擎 API 不会提醒你漏了边界情况，但领域模型会——状态机少了一个 transition、资源管理少了回滚策略，在领域层就能发现
- 领域模型是评判后续架构设计质量的标尺——没有它，架构设计没有参照基准

## 工作流

### 1. 读取关键文档

读取 `{task_dir}/.work/requirements.md` 全文。这是分析用户输入和深度采访后，完整的需求设计。文档结构：

- **关联功能系统** — 本次涉及哪些功能系统（新增/改造），已经按玩家可感知的系统做了初步分组
- **新增行为** — 玩家可见的行为清单，每条可验证
- **边界规则** — 已识别的边界情况和预期行为

通读后从整体理解：这个 feat 要做什么、涉及哪些功能系统、行为之间的关联关系。

读取 `{task_dir}/.work/grill-interview.md`，这是对用户输入的深度采访，深度挖掘的用户的诉求。从中提取与领域设计相关的约束。

同时读取 `{dev_dir}/requirements.md`（如存在），这是项目级需求设计，获取上下文来挖掘与其他模块的边界与交互，无交互模块则无视。

### 2. 划分功能模块

基于对需求文档全文的理解，将本次需求划分为**可独立建设的模块**。判断标准：这个模块能不能作为一个完整功能单元独立交付、独立验证？

```
需求文档描述:
  关联功能系统: 武器系统（新增）
  新增行为:
    1. 玩家按 R 键 → 进入换弹流程
    2. 换弹期间 → 射击键无效
    3. 空弹时按射击键 → 自动触发换弹
    4. 换弹中切武器 → 取消换弹，不补充弹药
    5. HUD 实时显示弹夹/备弹数量

模块划分:
  行为 1~5 共同构成"换弹"这个完整功能——弹夹管理、状态锁定、
  射击抑制、UI 刷新都是换弹的不同侧面，无法各自独立交付,
  → 一个模块: 换弹
```

**要求：**
- 划分依据是**可独立建设、独立验证**，不是按领域模式拆分
- 模块数量通常 1~5 个,划分更多不是完全禁止,是要给出合理理由,证明确实这些模块必须独立建设,理由要写在模块划分的后面
- 一个行为只能归属一个模块
- 模块不涉及文件/引擎构造

**防遗漏自查：** 用需求文档的"新增行为"清单逐条核对——每条行为能否归属到某个模块？有没有行为找不到归属？

### 3. 逐模块做领域分析

对每个模块，确定**用什么领域模型来建设**，完成三件套：

**这是什么模式：** 这个模块适合用哪种领域模式来建设？状态机 / 资源管理 / 事件队列 / 空间查询 / 命令模式 / 观察者 / 数据驱动 / ...

同一个模式可以出现在不同模块——两个模块各自独立使用状态机是完全正常的，不代表它们要合并。

**业界通用做法：** 这个模式在游戏开发中一般怎么实现。讲清楚核心概念、状态/阶段、关键规则。用引擎无关的语言。

**边界情况清单：** 至少覆盖：输入为空、容量上限、并发冲突、中断/取消。

**要求：**
- 每个模块独立完成三件套
- 领域描述不出现具体技术
- 如果某个模块的领域模式很明显且简单（如"显示一个静态文本"），简写即可

### 4. 写入

按 `references/domain-design-format.md` 的模板写入 `{task_dir}/.work/domain-design.md`。

```bash
mkdir -p {task_dir}/.work
```

## Completion Gate

永远不要声称任务完成，除非：

1. `{task_dir}/.work/requirements.md` 已读取并完整阅读理解内容
2. 功能模块已划分，每个行为的归属明确
3. 每个模块已完成领域分析三件套（模式、业界做法、边界清单）
4. 领域描述不含具体技术
5. `{task_dir}/.work/domain-design.md` 已按格式规范写入
6. 输出摘要

**输出摘要格式：**

```
## 领域建模完成

**文件：** {task_dir}/.work/domain-design.md

**功能模块：** {N} 个
  - {模块 A} → {模式}
  - {模块 B} → {模式}

**边界情况：** {N} 条
```
