# architecture.md 格式规范

`{dev_dir}/architecture.md` 是项目级架构文档，由 `game-dev:architecture` skill 创建和维护。plan 阶段读取它获取架构约束，exec 阶段的 coding agent 读取它了解已有架构决策。

## 核心原则

- **当前状态，不是变更日志。** 不写 `feat-3 新增了...` 或 `最后更新: feat-5`。文档始终反映项目架构的**当前状态**。
- **按子系统分级组织。** 同子系统的内容聚合在一个 heading 下，不分散在文件不同位置。
- **领域模型 + 引擎映射 + 关键约定 缺一不可。** 这三者是对 coding agent 最有价值的参考信息。
- **简洁可扫描。** 每个子系统节控制在一屏以内。coding agent 需要快速定位相关信息。

## 文件位置

```
{dev_dir}/architecture.md
```

`{dev_dir}` 从 `${CLAUDE_PLUGIN_ROOT}/references/{tech}/config.md` 的 `## 产物目录` 节获取。

## 模板

```markdown
# Architecture: {项目名称}

## Overall Architecture

### 子系统地图

{子系统间的关系图——Mermaid flowchart 或文字列表。描述各子系统的职责和数据流方向。}

### 数据流

{数据如何在子系统间流动——信号/事件总线/直接调用/持久化层。关键接口和契约。}

---

## Subsystem: {子系统名称}

### 领域模型

{这个子系统在游戏中本质上是什么——引擎无关的描述。领域概念、核心实体、它们之间的关系。}

**模式：** {状态机 / 资源管理 / 事件队列 / 空间查询 / 观察者 / 数据驱动 / ...}

**核心概念：**
- {概念 1} — {一句话说明}
- {概念 2} — {一句话说明}

**边界规则：**
| # | 边界情况 | 预期行为 |
|---|---------|---------|
| 1 | {情况} | {行为} |

### 引擎映射

{领域模型中的概念如何映射到引擎构造——场景节点、Resource、信号、autoload 等。}

**文件组织：**
| 类型 | 路径 | 说明 |
|------|------|------|
| 脚本 | `{scripts_path}/{subsystem}/` | {职责} |
| 场景 | `{scenes_path}/{subsystem}/` | {职责} |
| 资源 | `{resources_path}/{subsystem}/` | {职责} |

**核心构造：**
- {引擎构造 1} → 实现 {领域概念 1}
- {引擎构造 2} → 实现 {领域概念 2}

**对外接口：**
- {接口签名/信号/group 名称} — {约定说明}

### 关键约定

- {约定 1}：{说明}
- {约定 2}：{说明}

#### Feature: {子特性名称}

##### 领域增量
{这个子特性在领域模型上增加了什么}

##### 引擎增量
{实现上增加了什么文件/构造，改了什么，没改什么}

##### 新增约定
- {新增的约定}
```

## 章节命名约定

| 层级 | 格式 | 用途 |
|------|------|------|
| 一级 | `# Architecture: {项目名}` | 文档标题 |
| 二级 | `## Overall Architecture` | 总体架构（子系统地图 + 数据流） |
| 二级 | `## Subsystem: {子系统名}` | 子系统章节（领域模型 + 引擎映射 + 约定） |
| 三级 | `### 领域模型` / `### 引擎映射` / `### 关键约定` | 子系统内的固定三要素 |
| 四级 | `#### Feature: {子特性名}` | 子系统下的功能扩展 |

## 内容规则

**必须遵守：**
- 不写 feat-N、refactor-N 等任务编号
- 不写日期、作者、变更日志
- 同子系统的内容聚合在同一 `## Subsystem` 下
- 领域模型用引擎无关的语言描述（不出现 `Node`、`Signal`、`@export` 等引擎术语）
- 引擎映射必须具体到文件路径和构造名称

**禁止内容：**
- `TODO`、`FIXME` 等占位符
- "详见 xxx 文档"的引用（自包含，不依赖外部文档）
- 实现细节（如具体算法、变量名）
- 已被废弃的旧架构描述

## 更新规则

`--update` 模式合并时：

| 变更类型 | 合并操作 |
|---------|---------|
| 新子系统 | 追加到 Overall Architecture 的子系统地图，新建 `## Subsystem: {Name}` 节 |
| 架构变更 | 定位 `## Subsystem: {Name}` 节，替换/合并变更的部分。保留未变更部分。 |
| 功能扩展 | 定位 `## Subsystem: {Name}` 节，在末尾追加 `#### Feature: {Name}` 子节 |
| 无架构影响 | 不修改 |
