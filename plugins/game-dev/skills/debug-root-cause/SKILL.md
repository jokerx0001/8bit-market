# Debug Root Cause — BUG 根因分析

## The Iron Law

```
NO DIAGNOSIS WITHOUT BACKWARD TRACING FIRST
```

如果你没有逐层追踪到"正确输入→错误输出"的转换点，你不能声称找到了根因。猜的根因 = 没有根因。

## 输入

| 输入 | 说明 |
|------|------|
| BUG 描述 | 用户原始报告，原样引用 |
| 预期行为列表 | 用户确认的正确行为，逐条列出，含验证方式（`behavior` 或 `screenshot: 问题描述`）。screenshot testcase 通过命名约定 `test_{描述}_screenshot` 在测试目录中定位 |
| BUG 复现测试路径 | test agent 编写的失败测试文件（GUT）。screenshot 脚本从同目录 `visual/` 子目录下按命名约定发现 |
| before_attempts | `{task_dir}/.work/fix-attempts.md` 的路径。存在则读取——已验证为错误的路径，诊断时避开 |
| task_dir | 产出写入 `{task_dir}/.work/debug-analysis.md` |

## 流程

### 步骤 1：确认 BUG 存在

**GUT 测试：** 跑 BUG 复现测试，捕获确切的失败输出：

```bash
{test_runner} {test_path}
```

记录：
- 哪个 assertion 失败了
- actual vs expected 值是什么
- 完整的 traceback（如有）

**Screenshot 测试（如有）：** 按 `${CLAUDE_PLUGIN_ROOT}/references/{tech}/screenshot.md` 的 CLI 命令执行截图脚本，base64 解码保存到 `{task_dir}/.work/screenshots/{testcase_name}.png`。读对应 `.question` 文件，调用 `Skill("game-dev:visual-qa")`，将截图路径和问题内容传入 `$ARGUMENTS`。记录 visual-qa 返回的 `### Answer`。

**综合判定：** 任一测试 PASS（BUG 不存在）→ 报告给 fix-conductor，停止。可能是用户环境问题或 BUG 已被修复。

### 步骤 2：逆向追踪

**如 `before_attempts` 路径对应的文件存在：** 先读取——这些是已验证为错误的根因假设和修复路径。诊断时主动避开这些方向，从新的角度追踪。

**核心原则：永远不在症状点修。从失败点出发，逐层往上追，直到找到"正确输入变成错误输出"的转换点。**

```
失败点（测试 assertion 失败 / traceback 异常）
    ↓
问：什么代码直接产生了这个错误？
    ↓ 读这段代码
问：这段代码接收了什么输入？输入正确吗？
    ↓
    ├── 输入正确，输出错误 → 根因在这一层，停止追踪
    └── 输入本身就是错的 → 往上追一层：谁产生了这个错误输入？
                              ↓
                              重复
```

**每层只做一个判断：给定输入，这段代码的输出正确吗？**

- 输出正确 → 问题在上游，继续追
- 输出不正确 → 这一层就是根因位置

**追踪技巧：**

- 从 traceback 的最后一帧（最深层）开始，逐帧往上读调用链
- 对每个函数/方法：看它接收的参数 → 看它的逻辑 → 判断输出是否正确
- 数据流问题（值不对）：追谁设置了这个值 → 谁传给了它 → 一直追到赋值源头
- 控制流问题（走到了错误分支）：追哪个条件判断导致了错误分支 → 那个条件为什么为真/假

**阅读代码时：**
- 不只读报错的那一行，读完整的函数/方法
- 关注变量赋值、条件分支、循环边界
- 如果调用链跨文件，用 Grep 追踪符号引用

### 步骤 3：形成假设

追踪到根因位置后，用一句话表述：

```
"根因是 {代码位置} 中的 {具体逻辑/数据问题}，因为 {因果链}"
```

**好假设的特征：**
- 精确到具体代码行或逻辑表达式
- 能解释所有症状（包括用户报告的和测试捕获的）
- 能解释为什么预期行为没有发生

**假设不成立时：**
- 回到步骤 2，从新的角度重新追踪
- 检查是否有遗漏的调用路径或数据来源
- 最多 3 轮追踪，仍无结论则输出已知信息，请求 fix-conductor 人工介入

### 步骤 4：最小验证

**用最小的临时修改验证假设，确认后立即撤销。**

```
1. 做最小修改（改一行、加一个条件、修一个赋值）
2. 跑 BUG 复现测试（GUT + Screenshot 均执行，方法同步步骤 1）
3. 全部 PASS → 根因确认
   任一 FAIL → 假设错误，回到步骤 2
4. 撤销临时修改（git checkout / 手动还原）
```

**验证修改是 hack——不要把它写成正式修复。** 它的唯一目的是确认"改这里，BUG 就消失"。

### 步骤 5：写入 debug-analysis.md

保存到 `{task_dir}/.work/debug-analysis.md`：

```markdown
# 调试分析

## BUG 描述
{用户报告的 BUG，原样引用}

## 预期行为
{fix-conductor 阶段 1 确认的正确行为，逐条列出，含验证方式}
1. {行为 1}  — 验证方式: {behavior | screenshot: 问题描述}
2. {行为 2}  — 验证方式: {behavior | screenshot: 问题描述}
3. ...

## 根因
{经步骤 4 验证的根因，精确到具体代码位置 + 因果链}

## 证据链
1. {证据 1：BUG 复现测试的失败输出（GUT 具体 assertion 失败 + Screenshot visual-qa 结论，如有）}
2. {证据 2：逆向追踪路径 — 从症状到根因的每一层}
3. {证据 3：最小验证结果 — 临时修改了什么，GUT + Screenshot 从 FAIL → PASS}
4. {证据 4：排除的替代假设（如有）}

## 影响范围
| 文件 | 问题 |
|------|------|
| xxx | {具体问题描述} |

## 修复方向
{修复方案的概要描述，一两句话。不写实现代码。}
```

## Red Flags — 停下来，回到步骤 2

如果你发现自己在想：

- "看代码就知道问题在哪了，不用一步步追"
- "这个 BUG 很简单，直接写结论吧"
- "很可能就是 X 的问题，先写了再说"
- "追踪太慢了，我先猜一个再验证"
- "traceback 已经很清楚了，不需要逐层读"
- "这里不太对，但说不上来为什么——先跳过"
- 还没跑测试就开始写 debug-analysis.md
- 步骤 2 没追完就开始写根因

**以上任一条出现 → STOP。回到步骤 2，从失败点重新追。**

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| "BUG 很简单，不需要完整追踪" | 简单 BUG 也有根因。跳步 = 猜。 |
| "traceback 已经告诉我在哪了" | traceback 告诉你症状在哪，不是根因在哪。 |
| "我见过类似的，我知道原因" | 类似不是相同。每个 BUG 用自己的证据链说话。 |
| "来不及追踪了，先给结论" | 猜的根因会浪费 fix-conductor 更多时间。 |
| "测一下就能确认，不用读代码" | 测试告诉你"哪里不对"，代码告诉你"为什么不对"。两者都要。 |

## 找不到根因时

如果 3 轮追踪后仍无法确认根因：

1. 你已经完成了流程——不是你偷懒
2. 输出已知信息：追踪到了哪一层、卡在哪一层、为什么卡住
3. 标注不确定性，写入 `.work/debug-analysis.md`

**但：** 大部分"找不到"的情况是步骤 2 追得不够深。卡住之前，先检查：是否读完了每个相关函数的完整代码？是否追到了赋值源头？

## 硬约束

- **不实施修复。** 步骤 4 的临时修改必须撤销。debug-analysis.md 不含实现代码。
- **不修改测试文件。** BUG 复现测试由 test agent 管理。
- **测试 PASS 不代表没有 BUG。** 步骤 1 测试 PASS 时，检查测试是否真正覆盖了 BUG 场景。
- **最多 3 轮追踪。** 3 轮仍无结论 → 输出已知信息，标注不确定性，交 fix-conductor 决策。
- **不猜测。** "可能是 X" 不写入 debug-analysis.md——只记录已验证的结论。
