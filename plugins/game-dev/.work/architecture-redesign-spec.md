# Architecture.md 重构规格说明

## 问题诊断

当前 `architecture.md` 模板（`skills/plan/SKILL.md` 步骤 5）产出的文档缺少架构设计最核心的部分——**分模块规约**。

现有结构：
```
✅ 总体架构图 (Mermaid)        — 模块划分
✅ 界面/场景划分表              — 模块清单
✅ 数据流                      — 过程视图
✅ 交互约定                    — 跨模块通信
✅ 关键设计决策                 — 决策记录
❌ 分模块详细说明              — 缺失！每个模块的职责、领域映射、技术方案、接口契约
```

对照 arc42 / C4 / SEI "Views and Beyond" 三个权威来源，架构文档必须回答三个问题：

| 问题 | 权威来源 | 当前状态 |
|------|---------|---------|
| 分几个模块？每个模块做什么？ | arc42 Building Block View / C4 Component / SEI Module View | 只有清单，无规格说明 |
| 模块间怎么交互？ | arc42 Runtime View / C4 Interactions / SEI C&C View | 有，但偏散 |
| 为什么这样做？ | arc42 Decisions / SEI Rationale | 有，较完整 |

**根因：总体架构图之后缺少一个 `## 模块规约` 章节**，它应是架构文档的脊梁骨。

---

## 改造目标

让 architecture.md 成为真正的模块级架构设计——以 domain-design.md 产出的模块为骨架，逐个模块讲清楚"领域来源 → 技术落地 → 接口契约 → 关键决策"。

三层递进关系保持不变，但每层的职责边界更清晰：

```
domain-design.md    →  WHAT：这是什么模式？有什么边界？（引擎无关）
architecture.md     →  HOW（模块级）：分几个模块？每个模块用什么技术？模块间怎么通信？（引擎映射）
design.md           →  HOW（代码级）：每个行为具体的数据结构、变量、时序、生命周期
```

## plan/SKILL.md 步骤 5 的配套修改

步骤 5（架构设计）的指令需要从当前的模板替换为上述新模板。具体改动点：

### 改动 1：步骤 5 输出的章节列表

当前 SKILL.md 的步骤 5 模板只有 4 节：
```
总体结构 / 界面场景划分 / 数据流 / 交互约定
```

需替换为 5 节：
```
总体架构 / 模块规约 / 运行时视图 / 跨模块约定 / 架构决策记录
```

### 改动 2：步骤 5 的方法描述

当前步骤 5 描述：
> 调用 Skill 工具加载 superpowers:brainstorming，基于领域模型做引擎层映射

需增强为：
- 首先从 domain-design.md 提取模块列表
- 对每个模块，调用 brainstorming 回答：领域概念 → 引擎构造 → 接口设计 → 关键取舍
- 每个模块的结论写入"模块规约"节的一小节

### 改动 3：步骤 6（任务拆分）的输入

当前步骤 6 从 architecture.md 的"界面/场景划分"表提取模块。改造后需改为从"模块规约"节提取——**每个 `### 2.N {模块名}` 对应一个 AI 任务**。

### 改动 4：无需改动的部分

- `plan-format.md` 无需修改——plan.md 输出给 exec 的格式不变
- `architecture-format.md`（项目级 architecture.md）保持现有结构——它是跨 feat 的聚合，不需要单 feat 级别的模块规约
- `.work/design.md` 保持不变——它仍然是代码级详细设计

## 对照：改造前后的区别

以 zombies-demo feat-6 为例：

### 改造前 architecture.md

```
总体结构 (Mermaid)

数据流 (文字)
交互约定 (文字)

```

### 改造后 architecture.md

```
1. 总体架构
   - 模块关系图 (Mermaid)
   - 模块职责一览表 (3 列：模块/职责/领域概念)

       领域来源 → 职责 → 技术映射 → 对外接口 → 行为契约 → 设计理由
   2.2 SpawnPoint 节点层
       领域来源 → 职责 → 技术映射 → 对外接口 → 行为契约(5 态状态机) → 设计理由
   2.3 主场景装配
       领域来源 → 职责 → 技术映射 → 对外接口 → 设计理由

3. 运行时视图
   - 序列图：正常刷新流程
   - 序列图：spawn_data=null 降级路径

4. 跨模块约定
   - .tres 数据驱动、duck-type 契约、Timer 节点惯例
   - 与现有架构的契合点

5. 架构决策记录
   7 条决策的表格化（增加"替代方案"列）
```
