# Refactor Conductor Skill

在主会话中协调对现有代码的重构和改造，通过 TDD 方式确保修改安全。

## 核心原则

**重构铁律：先有测试护栏，再改代码**

```
┌──────────────────────────────────────────────────────────┐
│  NO CODE MODIFICATION WITHOUT TEST COVERAGE FIRST       │
└──────────────────────────────────────────────────────────┘
```

## 与 tdd-conductor 的区别

| 维度 | tdd-conductor（新功能） | refactor-conductor（重构） |
|------|------------------------|---------------------------|
| 前提 | 无现有实现 | 有现有工作代码 |
| 前置工作 | 读设计文档，拆分任务 | 读现有源码，评估影响 |
| RED 策略 | 测试失败因为功能未实现 | 测试失败因为断言目标行为，但代码还是旧行为 |
| GREEN 策略 | 从零实现 | 修改现有代码 |
| 额外验证 | 无 | 必须运行所有已有测试确保不破坏 |

## 多 Agent 架构

```
┌─────────────────────────────────────────────────────────┐
│                   主会话 (你)                             │
│               遵循 refactor-conductor skill              │
│                                                          │
│   1. 分析现有代码                                         │
│   2. 评估影响范围                                         │
│   3. 生成变更计划 → 等待用户审批                           │
│   4. 协调 TDD 重构循环                                    │
│                                                          │
│          ┌──────────────┐    ┌──────────────┐            │
│          │  test agent  │    │ coding agent │            │
│          │  (RED: 测试) │    │ (GREEN: 改码)│            │
│          └──────────────┘    └──────────────┘            │
│              Agent 工具          Agent 工具               │
└─────────────────────────────────────────────────────────┘
```

## 工作流程

### 第一步：分析现有代码

读取用户引用的代码文件和相关模块，建立当前实现的完整认知：

1. 读取用户提到的参考文件
2. 使用 Glob/Grep 发现所有相关文件（同模块、同模式的代码）
3. 识别当前实现模式（方法签名、调用链、依赖关系）
4. 查找已有测试文件
5. **将分析报告写入 `<task_dir>/analysis.md`**

**写入 `analysis.md` 的内容：**
```markdown
# 现有代码分析

## 重构目标
{用户描述的重构目标，原样引用}

## 当前实现模式
- 模式描述: {当前代码是怎么做的}
- 涉及文件: {文件列表，含相对路径}

## 已有测试
- {已有测试文件列表，或"无已有测试"}

## 依赖关系
- {调用链和依赖}
```

### 第一步 b：检测技术栈

在分析完现有代码后，调用 `stack-detector` skill：
- 入参：`task_dir = .neonbit-vibe-factory/refactor-{N}/`
- 等待用户确认门通过
- 完成后 `refactor-{N}/stack.json` 与 `refactor-{N}/routing-table.md` 已落盘

### 第二步：影响评估

列出所有需要修改的文件和方法，按风险分级，**将影响评估写入 `<task_dir>/impact.md`**：

**写入 `impact.md` 的内容：**
```markdown
# 影响评估

## 需要修改的文件
| 文件 | 方法/位置 | 变更类型 | 风险 |
|------|-----------|----------|------|
| XxxController.java | all methods | 参数替换 | 低 |
| XxxService.java | createXxx() | 签名变更 | 中 |

## 潜在风险
- {可能破坏的功能}
- {跨模块影响}

## 确认无需修改
- {排除的文件及排除原因}
```

### 第三步：变更计划（等待用户审批）

**第三步 a：加载测试约束**

在拆分任务之前，必须先读取测试相关的 rules，了解哪些文件/类型不需要测试：

1. 读取 `<task_dir>/routing-table.md`
2. 提取所有 `Applies to` 列包含 `test` 的 rules 文件路径
3. **Read 这些文件**，特别关注其中的排除约束（如"不写 DTO/Record/VO/Config/Mapper/Entity 的测试"、"只对 Service 和 Util 编写测试"等）
4. 将提取的约束记录下来，作为任务拆分的过滤条件

**第三步 b：拆分任务**

将变更拆分为独立的 TDD 任务，每个任务对应一个可验证的修改单元。**拆分时必须遵守第三步 a 中加载的测试约束**，不为被明确排除的文件类型生成测试任务。

**将变更计划写入 `<task_dir>/change-plan.md`**：

```markdown
# 变更计划

## 任务 1: XxxController - {变更描述}
- 文件: XxxController.java
- 变更: {具体修改内容}
- 测试策略: {集成测试/单元测试}

## 任务 2: ...

## 特殊约束
- {用户指定的约束，如"禁止mock"等}

## 测试排除项（基于 rules）
- {从 test rules 中提取的排除约束，如"不测试 DTO/Entity/Config/Mapper"}
```

**必须等待用户明确批准后才能继续。** 用户可以要求修改计划（修改后重新写入 `change-plan.md`）。

### 第三步 b：上下文清理

用户批准变更计划后，输出上下文清理声明，进入 TDD 阶段：

```
## 上下文已清理

设计文档已就绪，TDD 重构阶段将以设计文档为准：
- analysis.md — 现有代码分析
- impact.md — 影响评估
- change-plan.md — 变更计划（已审批）

不参考任何之前的分析讨论、方案辩论等过程性内容。
设计文档与之前讨论冲突时，以设计文档为准。
```

### 第四步：TDD 重构循环

**前置：读取设计文档**

在开始 TDD 循环前，重新读取以下文档作为唯一真相源：
- `<task_dir>/analysis.md` — 现有代码分析
- `<task_dir>/impact.md` — 影响评估
- `<task_dir>/change-plan.md` — 变更计划

**约束（必须遵守）**：
- 只参考设计文档：analysis.md、impact.md、change-plan.md
- 不参考过程讨论：不参考分析阶段的讨论、方案辩论等任何过程性内容
- 设计文档与之前讨论冲突时，以设计文档为准

对变更计划中的每个任务，执行 RED → GREEN → REFACTOR 循环。

#### 4a. RED 阶段 — spawn test agent

**注入 rules**：从 `<task_dir>/routing-table.md` 中提取 test 角色对应的 rules 路径列表，追加到 prompt 的"必读编程规范"段：

```
## 必读编程规范（第零步强制 Read）

以下文件必须在开始任务前全部读取：
- {path1}
- {path2}
- ...
```

```
Agent({
  subagent_type: "neonbit-vibe-factory:test:test",
  prompt: `
## 任务描述
为 {目标} 编写/修改测试以验证重构后的目标行为

## 重构上下文
当前行为: {当前代码的行为描述}
目标行为: {重构后期望的行为描述}

## 测试策略
编写测试验证 TARGET（目标）行为：
- 测试断言重构后的预期行为
- 测试预期会失败，因为当前代码还是旧行为
- 如果已有测试文件，在其基础上修改或追加

## 现有代码位置
{实现文件路径}

## 测试文件位置
{测试文件路径}

## 约束
1. 测试验证的是目标行为（重构后的预期）
2. 测试预期会失败（因为代码还没改）
3. 测试名称清晰描述目标行为
4. 不编写任何实现代码

开始执行 RED 阶段。
  `
})
```

#### 4b. 评估 RED 返回结果

**检查点：**
1. 测试文件已创建/修改？
2. 测试断言的是目标行为（不是旧行为）？
3. 测试失败原因正确（旧行为 vs 目标行为的差异）？
4. 测试代码语法正确？

```
├── 满足所有检查点 → 进入 GREEN 阶段
├── 不满足 → 记录具体缺口，重新 spawn test agent（最多 3 轮）
└── 超过 3 轮 → 标记需人工介入
```

#### 4c. GREEN 阶段 — spawn coding agent

**注入 rules**：从 `<task_dir>/routing-table.md` 中提取 coding 角色对应的 rules 路径列表，追加到 prompt 的"必读编程规范"段：

```
## 必读编程规范（第零步强制 Read）

以下文件必须在开始任务前全部读取：
- {path1}
- {path2}
- ...
```

```
Agent({
  subagent_type: "neonbit-vibe-factory:coding:coding",
  prompt: `
## 任务描述
修改 {目标} 的实现，使测试通过

## 重构上下文
当前行为: {当前行为描述}
目标行为: {目标行为描述}

## 测试文件位置
{测试文件路径}

## 实现文件位置
{实现文件路径}

## 实现约束
1. 修改现有代码使测试通过
2. 不修改任何测试代码
3. 不破坏其他现有功能
4. 不写假代码 (NotImplementedException / TODO)
5. 确保编译通过
6. 只修改变更计划中指定的范围

开始执行 GREEN 阶段。
  `
})
```

#### 4d. 评估 GREEN 返回结果

- 代码是否已修改（无 NotImplementedException/TODO）？
- 是否只修改了计划内的文件？
- 编译是否通过？

#### 4e. spawn test agent 验证

```
Agent({
  subagent_type: "neonbit-vibe-factory:test:test",
  prompt: `
## 任务描述
运行测试，确认所有测试通过

## 模式
GREEN

## 测试文件位置
{测试文件路径}

## 验证要求
1. 运行目标测试，确保新/修改的测试通过
2. 运行同模块的所有已有测试，确保没有破坏
3. 如果有测试失败，报告失败的测试和原因

开始执行测试验证。
  `
})
```

#### 4f. 评估测试验证结果

```
├── 所有测试通过（包括已有测试） → 进入 REFACTOR 阶段
├── 新测试失败 → 将失败信息反馈给 coding agent，重新实现（最多 3 轮）
├── 已有测试被破坏 → 严重问题，将破坏信息反馈给 coding agent 修复
└── 超过 3 轮 → 标记需人工介入
```

#### 4g. REFACTOR 审查

主会话直接审查代码质量：

**审查清单：**
1. 修改是否符合变更计划的范围？
2. 是否有超出计划的改动？
3. 是否修改了测试代码？（不允许）
4. 是否有空代码或假代码？
5. 代码可读性和命名是否清晰？
6. 所有已有测试是否仍然通过？

**如果代码不合格：** 记录问题，spawn coding agent 修复，重新审查。
**如果代码合格：** 标记任务完成，继续下一个任务。

### 第五步：重复循环

对变更计划中的每个任务重复第四步，直到全部完成。

## 状态追踪

每完成一个任务输出状态报告：

```
## 重构状态报告

### 当前进度
重构 - 任务 2/5

### 任务状态
| 任务 | 状态 | RED | GREEN | REFACTOR |
|------|------|-----|-------|----------|
| 1. UserController | 完成 | done | done | done |
| 2. OrderController | 进行中 | done | 进行中 | - |
| 3. FileController | 待分配 | - | - | - |

### 问题追踪
| 问题 | 状态 | 说明 |
|------|------|------|
| 无 | - | - |
```

## 约束（绝对不允许违反）

| 约束 | 说明 |
|------|------|
| 测试先行 | 没有测试覆盖就不能修改代码 |
| 不改测试 | coding agent 不允许修改测试代码 |
| 范围纪律 | 只修改变更计划中列出的文件 |
| 已有测试不破坏 | 所有已有测试必须继续通过 |
| 真实代码 | 不允许空代码、假代码 (NotImplementedException) |
| 全部 green | 有任何测试失败就不能算完成 |
| 人工兜底 | 某个任务 3 轮失败后标记人工介入，继续下一个 |

## 错误处理

| 问题 | 处理 |
|------|------|
| test agent 写了断言旧行为的测试 | 指出问题：测试应断言目标行为，重新 spawn |
| coding agent 改了测试 | 严重警告，重新 spawn coding agent |
| coding agent 修改了计划外的文件 | 指出超范围，重新 spawn coding agent |
| 已有测试被破坏 | 将破坏信息反馈给 coding agent，要求修复 |
| 重构后测试失败 | 撤销重构，保持测试 green |
